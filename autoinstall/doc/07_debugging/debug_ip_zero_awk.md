# Debug: error-commands IP Logged as 0.0.0.0

**Date:** 2026-03-30
**Target Node:** 10.99.236.91
**Symptom:** IP address logged as `10.99.236.0` (4th octet zero) in IPMI SEL during abort/error path

---

## 1. Issue Description

When an autoinstall fails and `error-commands` runs, the IPMI SEL receives marker `0x13` with the 4th octet as `0x00`, making the reported IP `x.x.x.0` instead of the actual address.

The `late-commands` IP logging on the same ISO reports the correct IP. Only the `error-commands` path is affected.

---

## 2. Root Cause Analysis

### Confirmed on 10.99.236.91 via live SSH session

The ISO at 10.99.236.91 was a **transitional build**: `late-commands` had the single-quoted awk fix applied, but `error-commands` still used the old double-quoted awk. This caused a 3-step failure:

### Step 1 — Double-quoted awk expands `$1` as a shell variable

```bash
# In user-data (generated by old build script):
IP=$(hostname -I | awk "{print $1}")
```

`$1` inside double quotes is expanded by the shell as positional parameter 1 (empty in this context).
Awk receives `{print }` which prints `$0` — the **entire line**:

```
IP = "10.99.236.91 192.168.90.130 172.17.0.1"
```

### Step 2 — Nested double quotes break the eval/awk line

```bash
eval $(echo "$IP" | awk -F. "{printf \"o1=%s; o2=%s; o3=%s; o4=%s\", $1, $2, $3, $4}")
```

After shell parsing, `"` inside the awk program creates unbalanced quoting.
The multi-IP string `10.99.236.91 192.168.90.130 172.17.0.1` split by `.` gives awk fields like `91 192` for field 4. The eval output includes bare tokens like `192` which bash tries to execute as a command:

```
bash: 192: command not found
```

`o4` is never assigned.

### Step 3 — Empty `o4` produces 0x00

```bash
h4=$(printf "0x%02x" "$o4" 2>/dev/null || echo "0x00")
```

`printf "%02x" ""` → treats empty as `0` → `h4=0x00`

IPMI marker `0x13 0xEC 0x00` → decoded as IP `10.99.236.0`.

### Contrast: late-commands (correct)

```bash
HOST_IP=$(hostname -I | awk '{print $1}')
eval $(echo "$HOST_IP" | awk -F. '{printf "o1=%s; o2=%s; o3=%s; o4=%s", $1, $2, $3, $4}')
```

Single-quoted awk → `$1` is an awk field variable (not a shell var) → only the first IP is captured → all 4 octets correctly parsed.


---

## 4. Fix

The fix was already applied to the build script (`build-ubuntu-autoinstall-iso.sh`) prior to this debug session.

**error-commands section — before (buggy):**
```bash
IP=\$(hostname -I | awk "{print \$1}")
...
eval \$(echo "\$IP" | awk -F. "{printf \"o1=%s; o2=%s; o3=%s; o4=%s\", \$1, \$2, \$3, \$4}")
```

**error-commands section — after (fixed):**
```bash
IP=\$(hostname -I | awk '{print \$1}')
...
eval \$(echo "\$IP" | awk -F. '{printf "o1=%s; o2=%s; o3=%s; o4=%s", \$1, \$2, \$3, \$4}')
```

In the heredoc, `\$` → `$` and single quotes are literal → the generated user-data contains `'{print $1}'` where `$1` is preserved as an awk field variable, not expanded by the installer shell.

---

## 5. Resolution

The ISO on node 10.99.236.91 was built before the fix was applied. **Rebuild the ISO** with the current build script to get correct `error-commands` IP logging in future deployments.

No changes to `build-ubuntu-autoinstall-iso.sh` are needed — the fix is already in place.

---

## 6. Key Rule

> Always use **single-quoted** awk programs when embedding shell scripts inside a bash heredoc.
> Use `'{print \$1}'` in the heredoc → produces `'{print $1}'` in output → awk field variable (correct).
> Never use `"{print \$1}"` → produces `"{print $1}"` → bash expands `$1` at runtime (wrong).
